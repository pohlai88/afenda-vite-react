# Internationalization (i18n) guide (Afenda)

This document describes **planned** **i18n** for **`apps/web`**: typically **i18next** + **react-i18next**, stable keys, locale-aware dates, and API error code mapping. Legacy reference only: `.legacy/.../react-i18n`.

**Status:** **Planned** for **`apps/web`** — not in current `package.json`. Optional **`i18next-http-backend`** for lazy JSON; optional **`i18next-browser-languagedetector`** for client detection.

## How we use i18n

| Topic | Convention |
| --- | --- |
| **Bootstrap** | Initialize **`i18next`** once (dedicated **`i18n.ts`** / **`i18n.js`**) and **import it before** the app root mounts; wrap the tree in **`<Suspense>`** when using async loading / default suspense behavior ([`useTranslation`](https://react.i18next.com/latest/usetranslation-hook)) |
| **React** | **`initReactI18next`** + typically **`interpolation: { escapeValue: false }`** (React escapes DOM text) |
| **Routing** | Choose URL prefix vs query vs cookie; align with **React Router** ([React](./react.md)) |
| **Keys** | Stable **namespaced** keys (`erp:invoices.title` or `ns` + dotted key)—avoid raw English as keys unless you follow [key fallback](https://www.i18next.com/principles/fallback) deliberately |
| **Dates** | [date-fns](./date-fns.md) locales + **`Intl`** for numbers |
| **API errors** | Map [API](../API.md) `error.code` to translated strings |
| **RTL** | Plan layout if targeting RTL locales ([Design system](../DESIGN_SYSTEM.md)) |

---

## Red flags

- **Hard-coding** English in ERP surfaces that must ship multi-language.
- **Mixing** locale detection strategies without one documented approach.
- **Omitting `Suspense` or `ready`** when lazy-loading namespaces and **`useSuspense`** is on—can flash untranslated keys or trigger missing-key side effects.
