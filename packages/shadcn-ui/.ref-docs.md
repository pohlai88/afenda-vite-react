# 1. North star

Your target should be:

> **feature code expresses business meaning, never visual decisions**

So feature code should say:

```ts
severity: 'high'
status: 'failed'
tone: 'warning'
surface: 'muted'
density: 'compact'
```

and **never** say:

```ts
'bg-red-500 px-3 text-sm rounded-md'
```

That means the governed stack becomes:

1. **Truth/domain layer**
   invariant, allocation, settlement, reconciliation, authority, evidence

2. **Semantic UI layer**
   badge, alert, panel, field, section, stat, table, timeline, evidence

3. **Token layer**
   colors, radii, spacing, typography, motion, elevation

4. **AST enforcement layer**
   blocks direct styling and blocks local semantic invention

# 8. Full AST rule set — final enterprise version

Your uploaded checker already has strong foundations around import checking, raw palette detection, inline styles, semantic map detection, and local variant factories. The final set should expand that into a true platform contract. 

I would define the rules in these groups.

---

## Group A — import governance

## `UIX-AST-IMPORT-001`

**Direct Radix import outside governed UI package**

Already present conceptually. Keep it. 

### Detect

* `@radix-ui/react-*`

### Why

Only wrapped primitives are allowed.

---

## `UIX-AST-IMPORT-002`

**`class-variance-authority` import outside governed UI package**

Already present. Keep it. 

---

## `UIX-AST-IMPORT-003`

**Tailwind helper import outside governed UI package**

### Detect

* imports of any local styling helper not explicitly approved
* examples:

  * `clsx`
  * `tailwind-merge`
  * local `cn` variants from non-governed packages

### Why

Features should not compose design language manually.

---

## `UIX-AST-IMPORT-004`

**Semantic adapter must come from governed package**

### Detect

Any import that looks like:

* `status map`
* `badge map`
* `severity map`
* `variant map`

from local feature files

### Why

Domain-to-UI mapping must come from `@afenda/ui/semantic` only.

---

## Group B — raw styling prohibition

## `UIX-AST-COLOR-001`

**Raw Tailwind palette class**

Already present. Keep it.  

Examples:

* `bg-red-500`
* `text-zinc-700`

---

## `UIX-AST-COLOR-002`

**Hardcoded color literal**

Already present. Keep it.  

Examples:

* `#ff0000`
* `rgb(255,0,0)`
* `hsl(...)`

---

## `UIX-AST-CLASS-001`

**Arbitrary Tailwind value**

Already present. Keep it.  

Examples:

* `p-[13px]`
* `rounded-[11px]`

---

## `UIX-AST-STYLE-001`

**Inline visual style prop**

Already present. Keep it.  

Examples:

* `style={{ color: ..., padding: ... }}`

---

# Group C — semantic invention prohibition

## `UIX-AST-SEMANTIC-001`

**Local semantic/style map detected**

Already present, but strengthen it. Right now it is partly name-based and heuristic-based.  

### Upgrade

Do not rely only on variable name patterns. Also detect:

* object literals indexed by domain status values
* object lookup used in JSX className/tone/variant selection
* ternaries that branch on status/severity and return visual values

---

## `UIX-AST-VARIANT-001`

**Local variant factory outside governed UI**

Already present. Keep it.  

### Detect

* `cva(...)`
* `get...Classes`
* `create...Variants`

---

# Group D — missing semantic usage

These are the new critical rules.

## `UIX-AST-SEMANTIC-REQUIRED-001`

**Raw `className` not allowed in feature UI**

### Detect

In governed product feature paths:

* JSX with `className`
* containing Tailwind utility tokens
* unless file is in approved UI owner package

### Why

Feature code should express semantics, not visual utilities.

### Allowed exceptions

* layout shell primitives
* temporary migration allowlist
* generated or vendor files

---

## `UIX-AST-SEMANTIC-REQUIRED-002`

**Domain status/severity rendered without governed semantic adapter**

### Detect

A prop or variable named like:

* `status`
* `severity`
* `state`
* `tone`
* `variant`

used directly in rendering logic without passing through:

* `getInvariantUiModel`
* `getAllocationUiModel`
* `getSettlementUiModel`
* equivalent governed adapter

### Why

This is how you enforce truth-to-UI mapping.

---

## `UIX-AST-SEMANTIC-REQUIRED-003`

**Raw container block used where semantic component required**

### Detect

`div`, `section`, `aside`, `header` with:

* className utilities
* children count above threshold
* looks like card/panel/alert/stat/form block

### Why

Use `SemanticPanel`, `SemanticSection`, `SemanticAlert`, `SemanticField`.

---

# Group E — token discipline

## `UIX-AST-TOKEN-001`

**Spacing utility outside governed UI**

### Detect

* `p-*`
* `px-*`
* `py-*`
* `m-*`
* `gap-*`
* `space-*`

in feature code

### Why

Spacing rhythm must come from semantic components or approved layout primitives.

---

## `UIX-AST-TOKEN-002`

**Typography utility outside governed UI**

### Detect

* `text-xs/sm/lg/...`
* `font-*`
* `leading-*`
* `tracking-*`

in feature code

### Why

Typography should be governed by semantic text primitives.

---

## `UIX-AST-TOKEN-003`

**Radius/shadow/border utility outside governed UI**

### Detect

* `rounded-*`
* `shadow-*`
* `border-*`
* `ring-*`

in feature code

### Why

Surface treatment belongs to semantic panels and controls.

---

## `UIX-AST-TOKEN-004`

**Direct width/height sizing utility outside approved layout layer**

### Detect

* `w-*`
* `h-*`
* `min-w-*`
* `max-w-*`
* `grid-cols-*`

outside allowed layout files

### Why

Prevents silent divergence in layout rhythm.

---

# Group F — complexity and composition

## `UIX-AST-COMPLEXITY-001`

**Over-complex className composition**

### Detect

A `className` string with more than N utility tokens.

Suggested threshold:

* warning at 6
* error at 10

### Why

Long class strings in feature code are a drift smell even if not palette-related.

---

## `UIX-AST-COMPLEXITY-002`

**Nested conditional styling logic in render**

### Detect

Ternary chains, boolean conjunctions, or switch statements returning class names / tones / variants in JSX render blocks.

### Why

Visual decision trees belong in semantic adapters.

---

## `UIX-AST-COMPLEXITY-003`

**Multiple local wrapper components in feature file**

### Detect

Feature files defining several small UI-only subcomponents.

### Why

Usually means semantic UI is being rebuilt locally.

---

# Group G — truth mapping enforcement

## `UIX-AST-TRUTH-001`

**Invariant severity mapped locally to tone/icon/badge**

### Detect

Local status tables or conditionals where source values include:

* `low`
* `medium`
* `high`
* `critical`

and outputs are styling tokens, icons, badge variants, or alert variants.

### Why

Must use `getInvariantUiModel()`.

---

## `UIX-AST-TRUTH-002`

**Allocation/settlement/reconciliation state mapped locally**

### Detect

Keys like:

* `partial`
* `allocated`
* `reversed`
* `settled`
* `unmatched`
* `conflict`

mapped to UI outputs.

### Why

Must use governed domain adapters.

---

## `UIX-AST-TRUTH-003`

**Truth evidence state mapped locally**

### Detect

Keys like:

* `missing`
* `stale`
* `tampered`
* `unverified`

mapped to visual outputs.

### Why

This should become a signature AFENDA pattern, not ad hoc feature code.

---

# Group H — component boundary rules

## `UIX-AST-COMPONENT-001`

**Feature file imports low-level UI primitive instead of semantic wrapper**

### Detect

Imports of:

* Badge
* Alert
* Card
* Input groupings
* Table primitives

from low-level component packages when a semantic equivalent exists.

### Why

After semantic layer is ready, feature code should consume the semantic export first.

---

## `UIX-AST-COMPONENT-002`

**Feature file exports reusable visual component**

### Detect

A feature file exporting generic visual components like:

* `StatusBadge`
* `InfoPanel`
* `SectionCard`

### Why

Those belong in governed UI packages.

---

## `UIX-AST-COMPONENT-003`

**Feature-local icon-tone pairing**

### Detect

Feature file chooses icon + color/tone pairings locally.

### Why

Iconography should also be semanticized for repeated truth patterns.

# 10. Rollout strategy

Do not switch everything to hard fail on day 1.

## Phase 1

Fail immediately:

* raw colors
* direct radix
* cva outside ui
* inline styles
* local truth mapping

## Phase 2

Warn:

* raw className in features
* spacing/typography/radius utilities
* local wrappers

## Phase 3

Fail:

* feature `className`
* low-level primitive imports when semantic wrapper exists
* local block containers for panel/alert/stat

That rollout is safer and still converges toward full governance.
